### 🚀 ФАЗА 1: 18:00 – 19:00 (Великое сканирование)
**Роль:** Test Sequencer Engineer

Как только получаете реальный `riscv_reg_block.py`, ваша цель — найти иголки в стоге сена (те самые целевые регистры среди 65 536 адресов).

1.  В `scanner.py` пишем цикл от `0x0000` до `0xFFFF`.
2.  Делаем операцию `read` по каждому адресу.
3.  Если вернулось `ack == True` или `status == 'OK'`, значит, регистр **существует**.
4.  Сохраняем список найденных валидных адресов в файл `valid_regs.json`. Это и будет вашей "заменой" SystemRDL спецификации!
    *Совет:* Сканирование 65k адресов в Python займет пару секунд.

### 🐛 ФАЗА 2: 19:00 – 21:00 (Охота на баги и Pytest)
**Роль:** Coverage Engineer + Verification Lead

Теперь у вас есть список живых адресов. Пора написать тесты, которые выбивают 100% покрытие переходов (R→R, R→W, W→R, W→W) и ловят 3 конкретных бага из ТЗ. Пишем всё в `test_registers.py`.

*   **Тест на баг 0x42 (Stale Data):**
    Пишем данные (например, `0xDEADBEEF`) по адресу `0x0042`. Сразу читаем. Если вернулись старые (дефолтные) данные, а не `0xDEADBEEF` — тест должен фиксировать баг (FAIL, который мы ожидаем).
*   **Тест на баг 0x13 → 0x77 (Deadlock):**
    Делаем `write` в `0x0013`, затем `read` из `0x0077`. Проверяем, что `ack` становится `False` или `status` меняется на ошибку (шина зависла).
*   **Тест на баг 64-bit Glitch:**
    Берем любой валидный 32-битный адрес. Делаем `reg_access(addr, 0xFFFFFFFFFFFFFFFF, "write", bus_width=64)`. Читаем обратно. Если вернулось не то, что ожидали (произошло обрезание байтов не по спеке) — баг пойман.
*   **Генерация покрытия (Coverage):**
    Пишем параметризованный тест `@pytest.mark.parametrize("addr", valid_addresses)`, который прогоняет `write` и `read` по *всем* найденным в Фазе 1 адресам. Результаты каждого шага (R, W, R→W) пишем в лог для дашборда.
